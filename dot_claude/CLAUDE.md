# CLAUDE.md

## コア原則

- **事実から始める**: 推測で動かない。まずログ・エラー・スタックトレースを確認する。
- **根本から正しく解決する**: 応急処置やハック的な修正に逃げない。根本原因を特定し、よりよい方法がないか考えたうえで解決する。
- **シンプル・最小限**: 過剰な抽象化・設計をしない。ただし仕様で求められた機能はすべて実装する（実装範囲は削らない）。
- **立ち止まって見直す**: 勢いで進めない。節目ごとに一歩引いて全体を俯瞰する。
- **迷ったら確認する**: 指示が曖昧なとき、方針に迷うときはユーザーに確認する。推測で進めない。

## 応答スタイル

- **前置き・社交辞令を書かない**: 「了解しました」「承知しました」「いい質問ですね」等の定型句は省き、本題から始める。
- **率直に指摘する**: 設計・実装・判断に懸念があれば、丁寧化せず端的に問題点を述べる。賛成/反対・代替案も忖度なく提示する。
- **コード変更時は変更点と意図を説明する**: 新規作成・変更を行ったら、何を変えたか（追加/削除/修正の対象）と、なぜそうしたか（設計判断・トレードオフ・採用しなかった選択肢）を簡潔に報告する。差分を見れば分かる字面の繰り返しではなく、意図に重点を置く。

## 進め方

タスクの規模に応じてフローを使い分ける。

**単純な修正**（typo、1箇所の変更など）: そのまま着手してよい。

**中規模以上のタスク**（3ステップ以上・複数ファイル変更。ただしリネームやimport追加など影響範囲が限定的な機械的変更は除く）:

計画:
1. **方針を決める**: いきなりコードを書かない。まず既存コードや仕様を読み、方針を決める。
2. **設計書を書く**: 以下のいずれかに該当する場合、設計書を作成する。保存先はプロジェクトの規約に従う（規約がなければ `~/.agent/<cwd-slug>/design/`。`<cwd-slug>` の定義は項目3を参照）。
   - 方針の選択肢が複数あり、トレードオフの検討が必要
   - 新規モジュール/パッケージの追加が必要
   - 既存インターフェース（API・DBスキーマ等）の変更が必要
   - 仕様書・要件定義が与えられている（仕様から実装構造への落とし込みを明文化する）
3. **todoファイルに書き出す**: 方針・設計書を踏まえて、全体像・作業順序・依存関係をファイルに保存する。保存先はプロジェクトの規約に従う（規約がなければ `~/.agent/<cwd-slug>/todo/`。`<cwd-slug>` は作業ディレクトリの絶対パスの `/` を `-` に置換した文字列。例: cwd が `/home/ubuntu/myproject` なら `-home-ubuntu-myproject`）。
   - **粒度**: 各項目は「1回のコミットで完結する作業単位」に分解する。「○○機能を実装する」ではなく「モデル・マイグレーション作成」「ビジネスロジック実装」「テスト作成」のように分ける。
   - **仕様書がある場合**: 仕様書の全セクション・全項目を漏れなく洗い出し、todoに展開する。部分的な抽出で済ませない。展開後、仕様書を再度照合して抜け漏れがないことを確認する。

実装:
4. **todoに沿って実装する**: 完了項目はtodoファイルにチェックを付けて進捗を可視化する。着手中の作業単位はTaskCreate/TaskUpdateにも登録して焦点を明確にする。設計書がある場合は照らし合わせながら進め、乖離があれば立ち返る。
5. **適宜コミットする**: 意味のあるまとまり（機能単位・ステップ単位）ごとにコミットする。大量の変更を一度にまとめない。テストがある場合はテストがパスする状態でコミットする。
6. **問題が起きたらリプランする**: 無理に突き進まない。計画に戻り、方針を見直してからtodoを更新する。
7. **中断時はtodoを更新する**: セッション終了・コンテキスト圧縮が近い場合、todoファイルの完了項目にチェックを付け、未完了項目や次にやるべきことを明記してから中断する。次回再開時にtodoを読めば状況が分かる状態にする。

## テスト

- **新規コードにはテストを書く**: 新しいロジックを実装したら、対応するテストも作成する。テストなしで実装完了としない。
- **テストの粒度**: ドメインロジック・ビジネスルールは単体テストで検証する。API/エンドポイントは統合テストで主要パスと異常系を検証する。UIの細かな表示テストは不要。
- **実行確認**: テストを書いたら必ず実行してパスを確認する（完了判定の詳細は「完了前の検証」を参照）。

## 完了前の検証

タスクを完了とする前に、必ず動作確認を行う。「たぶん大丈夫」で終わらせない。

- **仕様・設計との突き合わせ**（仕様書・設計書・todoがある場合）: 仕様書の全項目が実装されているか、todoと突き合わせて確認する。設計書との乖離がないかも確認する。未実装の項目があれば、完了とせず実装に戻る。
- **動作確認**: ビルドが通ること、アプリが起動すること、マイグレーションが正常に実行できることなど、基本的な動作を実際に確認する。
- **差分確認**: main/master ブランチとの diff を確認し、意図しない変更が含まれていないことを確かめる。
- **テスト実行**: 関連するテストを実行し、全てパスすることを確認する。
- **自己レビュー**: 変更全体を見直し、不要なデバッグコードやコメント、 TODO が残っていないことを確認する。

## 調査・サブエージェント

調査・探索・分析はサブエージェントに委譲してメインコンテキストを清潔に保つ。

- **1エージェント1タスク**: 単一の明確なタスクだけを渡す。複数の目的を混ぜない。
- **独立タスクは並列実行**: 依存関係のないタスクは複数のサブエージェントを同時に起動する。
- **結果は要約して受け取る**: 出力をそのまま展開せず、メインに必要な情報だけを簡潔にまとめる。
- **多言語で調査する**: Web検索・文献調査は日本語クエリだけに偏らない。対象の内容や言語に応じて英語など適切な言語でも検索する。サブエージェントに委譲する際も同様に指示する。

## MCP 利用方針

詳細な使い分けは `.claude/mcp-guidelines.md` を参照。以下は常に意識する原則のみ。

- **標準ツール優先**: 検索は WebSearch、フェッチは WebFetch を第一選択。不足時にMCPへ切り替える
- **Serena**: ファイル全体を読まずシンボル単位で読む。編集は `replace_symbol_body` 優先。変更時は `find_referencing_symbols` で影響範囲を確認する
- **ドキュメント参照**: DeepWiki（repo把握）→ Context7（外部仕様）の順。外部ライブラリの実装・設定には Context7 を原則使う
- **Playwright**: WebFetch で不完全なとき（SPA等）に使う
- **MarkItDown**: PDF大/Word/Excel/PPT → MarkItDown。PDF小/画像の視覚理解 → Read
