# 作業手順書（workflow）

`~/.claude/CLAUDE.md` の「進め方」が定める**重い判断**タスクは、この手順書に沿って計画してから着手する。自明・軽い判断のタスクはこの手順を踏まず着手してよい。

## 計画

1. **方針を決める**: いきなり書かない。既存コード・仕様を読む。着手前に `~/.agent/<cwd-slug>/`（「作業ファイル」節参照）を `ls`／Glob で一覧し、既存設計書・todo を確認して、関連する未完ゴールがあれば再開する。
2. **設計書を書く（条件付き）**: 本質的な基準は「重い判断に当たるか（CLAUDE.md『進め方』参照）」。次のいずれかに当てはまり、かつ方針が自明でないなら作成する。
   - 方針に複数の選択肢があり、トレードオフ検討が必要（最重要・他はこれの代理指標）
   - 新規モジュール/パッケージの追加が必要
   - 既存インターフェース（API・DBスキーマ等）の変更が必要
   - 仕様書・要件定義が与えられており、規模・複雑さから事前の設計整理が要る

   いずれの条件に触れても、方針が一意に定まり検討の余地がなければ設計書は省く（「シンプル・最小限」を優先）。書く場合は目的・採用方針とその**理由**・不採用の代替案・主なリスクを書く。差分で分かる *what* より、後から追えない *why* を優先する。実装が設計と乖離したら、経緯と最終結論を追記して実態に合わせる。
3. **todo を書き出す（中断・再開や段取りが要るとき）**: 全体像・作業順序・依存関係を保存する（保存場所・命名は「作業ファイル」節）。各項目は「1コミットで完結する単位」に分解する（「○○機能を実装」ではなく「モデル・マイグレーション」「ビジネスロジック」「テスト」）。短く一気に終わるなら todo ファイルは省いてよい。仕様書がある場合は全セクション・全項目を漏れなく展開し、展開後に再照合して抜け漏れを確認する。設計書を作らないゴールでも、判断理由・前提・想定外の発見が生じたら todo 内に書き残す。

## 実装

4. **todo に沿って実装する**: 進捗の正本は常に todo ファイル。完了項目にチェックを付けて可視化する。TaskCreate/TaskUpdate は「今のセッションで手を動かすスライス」を可視化したいときに todo ファイルから派生させる一時ビューであり、第2の記録ではない。セッション終了時は todo ファイルに反映し、TaskCreate 側は破棄してよい。設計書があれば照らし合わせ、乖離があれば立ち返る。
5. **適宜コミットする**（git管理下）: 意味のあるまとまりごとに。大量変更を一度にまとめない。テストがあればパスする状態でコミットする。
6. **問題が起きたらリプランする**: 突き進まず計画に戻り、方針を見直して todo を更新する。
7. **中断時は todo を更新する**: 完了項目にチェックを付け、未完了項目と次の一手を明記してから中断する。次回 todo を読めば状況が分かる状態にする。

## 作業ファイル（設計書・todo）

特定タスクの計画・進捗・設計判断の置き場。プロジェクト固有の規約があればそれに従い、なければ以下に従う。

- **メモリ機構との住み分け**: 進行中タスクの手順・進捗・設計判断は `~/.agent` に置く。何をどちらに書くか（ハーネスのメモリ機構との住み分け）は CLAUDE.md「記憶と作業ファイル」が正本。
- **保存場所**: `~/.agent/<cwd-slug>/` 直下にフラットに置く。`<cwd-slug>` は作業ディレクトリの絶対パスの `/` を `-` に置換した文字列で、`pwd | sed 's#/#-#g'` で得られる（例: cwd が `/home/ubuntu/myproject` なら `-home-ubuntu-myproject`）。ディレクトリが無ければ初回書き込み時に `mkdir -p` で作成する。
- **ファイル名**: todo は `<YYYYMMDD>-<ゴール名>.todo.md`、設計書は `<YYYYMMDD>-<ゴール名>.design.md`（`<YYYYMMDD>` は作業開始日）。日付＋ゴール名が共通なので同一ゴールの資料は隣接し、ファイル名順が時系列順になる。
- **1ファイル = 1ゴール**: 1つの todo には一貫した1ゴールだけを書く。ドメイン・目的の異なる作業（例: インフラ移行とアプリ個別設定）を束ねない。これで設計書との範囲のズレを防ぐ。設計が必要なゴールは todo と設計書を**同じゴール名**でペアにし、種別はサフィックス（`.todo.md` / `.design.md`）で区別する。
- **参照は相対リンク**: ファイル間の参照（todo→設計書など）は絶対パスでなく相対リンクにする。

## テスト

テストを書く/書かないの判断・実行確認の原則は CLAUDE.md「テスト・完了前の検証」が正本。本節はその粒度の補足。

- **粒度**: ドメインロジック・ビジネスルールは単体テスト、API/エンドポイントは統合テストで主要パスと異常系を検証する。UIの細かな表示テストは不要。

## 完了前の検証

完了前に必ず動作確認する原則は CLAUDE.md「テスト・完了前の検証」が正本。以下はその具体手順。

- **計画との突き合わせ**: todo・設計書・仕様書があれば全項目が実装されているか突き合わせる。設計書が実態と一致しているかも確認する。未実装があれば実装に戻る。
- **動作確認**: 作業種別に応じた成果の確認を実際に行う。アプリ実装ならビルドが通る・起動する・マイグレーションが正常実行できる等。インフラ・設定なら適用・反映の確認、ドキュメント・調査なら成果物が要件を満たすかの確認。
- **差分確認**: git管理下なら main/master との diff を確認し、意図しない変更が混ざっていないか確かめる。git管理外なら、変更・新規作成したファイルを列挙し、意図しないファイルに触れていないか目視確認する。
- **テスト実行**: 関連テストを実行し全てパスを確認する。
- **自己レビュー**: 不要なデバッグコード・コメント・TODO が残っていないか確認する。
