以下の対象をコードレビューしてください。

## 対象
$ARGUMENTS

引数は以下のように解釈する。

- ファイルまたはディレクトリのパス → そのコードを対象とする（ディレクトリは配下の全ソースファイル）
- PR 番号（例: `123`, `#123`）→ `gh pr diff` で取得した差分を対象とする。`gh pr view` で PR の説明も読み、変更意図を把握する
- git のリビジョン範囲（例: `main..HEAD`, `HEAD~2..HEAD`）→ その範囲の差分を対象とする
- 引数がない場合 → working diff（未コミットの変更。なければ現在のブランチとベースブランチの差分）を対象とする。差分も存在しない場合は、ユーザーに対象を確認する

## チェック観点

### 1. 正しさ
- ロジックの誤り、仕様との不一致
- 境界条件・エッジケース（空・null・0件・上限・重複・タイムゾーンなど）の考慮漏れ
- エラー処理の漏れ、握りつぶし、リソースリーク
- 競合状態・並行処理の問題
- 後方互換性の破壊（公開 API・データ形式・設定の変更）

### 2. 設計・シンプルさ
- 過剰な抽象化、不要な複雑さ、YAGNI 違反
- 既存コード・既存ユーティリティの再利用漏れ（車輪の再発明）
- 重複コード、責務の混在
- 応急処置・ハックになっていて根本解決になっていない箇所
- 非推奨・古い API やパターンの使用（最新の推奨手法とのズレ）

### 3. セキュリティ
- インジェクション（SQL・コマンド・パストラバーサルなど）
- 認証・認可の漏れ、権限チェックの不備
- 秘密情報（鍵・トークン・パスワード）のハードコード、ログへの漏出
- 信頼できない入力の未検証、安全でないデシリアライズ

### 4. テスト・検証
- 新規・変更されたロジックに対するテストの欠落
- テストが仕様を検証できているか（実装をなぞるだけのテスト、アサーション不足）
- 見落とされているテストケース（異常系・境界値）

## 動作確認
コードを読むだけで終わらせず、対象の変更が実際に意図どおり動くかを確認する。

- 既存のテストスイートがあれば実行する（対象範囲に関連するテストだけで可）
- 実行可能なコードは実際に動かして確認する。CLI・スクリプトは代表的な入力とエッジケース（境界値・異常系）で実行する
- **UI を持つ web アプリ・フロントエンドは Playwright で確認する**: アプリを起動し、`browser_navigate` → `browser_snapshot` / `browser_click` 等で変更に関わる画面・操作を実際に実行する。表示崩れ・コンソールエラー（`browser_console_messages`）・ネットワークエラー（`browser_network_requests`）も確認する。必要に応じてスクリーンショットを撮って報告に添える
- **API サーバーのみの変更は curl 等でエンドポイントを叩いて確認する**（正常系と、変更に関わる異常系）
- コードリーディングで疑わしい箇所を見つけたら、可能な限り動作確認で再現を試みる。再現できた指摘には再現手順と結果を含める
- 動作確認は開発環境・ローカルに閉じて行う。本番環境・外部サービスへの書き込みを伴う確認はしない
- 環境構築が必要で実行できない場合は、無理に環境を作らず「動作確認できなかった範囲」として率直に報告する

## ビルトインスキルの併用
以下に該当する場合は、自分のレビューに加えてビルトインスキルを Skill ツールで呼び出し、その指摘を裏取りしたうえで本報告フォーマットに統合する（重複する指摘はまとめ、通し番号は統合後に振り直す）。該当しなければ呼ばない。

- **差分が大きい、またはバグの見落としが致命的な変更**（リリース直前、決済・データ移行など）→ `code-review` を併用し、網羅性を上げる
- **セキュリティ影響が大きい変更**（認証・認可、外部入力の処理、秘密情報の扱い、公開エンドポイントの追加）→ `security-review` を併用する

`code-review ultra`（クラウド多エージェントレビュー）は自分では起動できない。さらに深い網羅レビューが有効と判断した場合は、総評でユーザーに実行を提案する。
併用した場合は、総評にどのスキルを併用したかを記載する。

## 手順
1. 対象のコードまたは差分を正確に把握する。差分レビューの場合も差分だけで判断せず、変更された関数の呼び出し元・呼び出し先、関連する型定義・設定・テストを読み、文脈を踏まえて評価する
2. 「チェック観点」の4観点でコードをチェックする
3. 「動作確認」を行い、テスト結果・実行結果と疑わしい箇所の再現可否を確かめる
4. 「ビルトインスキルの併用」に該当すればビルトインスキルを呼び出し、指摘を統合する
5. レビュー結果の全詳細を「レビュー記録ファイル」に書き出す
6. チャットには「報告フォーマット」で要点を報告する

指摘は根拠を持てるものに絞る。推測にとどまる指摘は、推測であることを明示する。スタイルの好みだけの指摘はしない。

## レビュー記録ファイル
チャット報告とは別に、レビュー結果の正本となる記録ファイルを作成する。/fix-inspected はこのファイルを読んで修正し、対応状況を追記する。

- **保存場所**: `~/.agent/<cwd-slug>/` 直下（`<cwd-slug>` は `pwd | sed 's#/#-#g'` で得る。workflow スキルの「作業ファイル」と同じ規約）。ディレクトリが無ければ `mkdir -p` で作成する
- **ファイル名**: `<YYYYMMDD>-<対象名>.review.md`。`<対象名>` はレビュー対象を表す短いスラッグ（ブランチ名・PR番号・ディレクトリ名など）。同日同対象の再レビューは同じファイルを更新し、末尾に再レビューの節を追記する
- **詳細度**: チャット報告より詳細に書く。チャットに書く要点に加え、指摘ごとに以下を含める
  - 該当コードの引用（問題の箇所が分かる範囲）
  - 問題が起きるメカニズムの説明（どういう入力・状態・順序で、何がどう壊れるか）
  - 動作確認で再現した場合は、実行したコマンド・操作と実際の出力・観察結果
  - 具体的な修正案（方向性だけでなく、可能ならコードで示す）と、その修正で問題が解消する理由
- **記録するメタ情報**: 冒頭にレビュー日時、対象（コミットハッシュ・PR番号・パス）、実施した動作確認の一覧、併用したビルトインスキルを記載する
- **指摘番号**: チャット報告と同じ通し番号を使い、チャットとファイルで指摘が1対1に対応するようにする
- **対応状況欄**: 各指摘に「対応状況: 未対応」の行を設けておく（/fix-inspected が更新する）

## 報告フォーマット
チャット報告は要点に絞る。各指摘は問題点と修正の方向性が分かる最小限にとどめ、コードの引用や再現ログの全文は記録ファイル側に書く。
観点ごとにまとめ、指摘ごとに区切り線（---）で区切って報告すること。
各指摘には全観点を通じた通し番号を振ること（#1, #2, #3, ...）。
各指摘の先頭には該当するファイル名と行番号を `file_path:line_number` の形式で記載すること。
指摘のない観点はセクションごと省略する。

### 正しさ / 設計・シンプルさ / セキュリティ / テスト・検証（各観点共通）

---
#番号 `file:line` 【重要度: 高 / 中 / 低】
問題点: 何が問題か。バグの場合は、どういう入力・状態でどう壊れるかを具体的に述べる（動作確認で再現できた場合は再現手順と結果も記載する）
修正案: 修正の方向性（コードで示せる場合は短く示す）
---

重要度の目安:
- **高**: バグ・セキュリティ問題など、修正しないと誤動作・被害が起きるもの
- **中**: 将来の保守・変更で問題になる設計上の問題、テストの欠落
- **低**: 改善が望ましいが、放置しても実害が小さいもの

### 動作確認結果
何をどう確認したかと結果を記載する。

- **実行したこと**: テスト実行 / アプリ起動 / Playwright での操作 / curl など、実施した確認とその結果（パス・失敗・観察された挙動）
- **確認できなかった範囲**: 環境等の制約で動作確認できなかった箇所と理由（なければ省略）

### 良い点
設計や実装で特に優れている点があれば簡潔に挙げる（なければ省略）。

### サマリー
最後に以下を記載する。

- **件数**: 正しさ X件 / 設計・シンプルさ X件 / セキュリティ X件 / テスト・検証 X件（重要度 高 X件 / 中 X件 / 低 X件）
- **記録ファイル**: レビュー記録ファイルのパス
- **総評**: レビュー全体を通じた所見を1〜3文で述べる（マージ可否の見解、特に優先して直すべき点、併用したビルトインスキル、ultra を提案する場合はその旨）。

修正が必要な場合は `/fix-inspected` で指摘番号を指定して修正できることを最後に案内する。
