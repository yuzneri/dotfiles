Limitless MCP (`searchLifelogsWithTranscripts`) の癖と、それを踏まえた取得・解釈の手順。
transcript を扱うあらゆる作業で本 skill のルールに従う。
(セッション・登壇者・公開タイムテーブル等イベント関連の手順は本 skill には含めない。
必要なら呼び出し側のプロンプトで補う)

## 取得手順の基本

- **全ページネーション必須**: query なしでも API は startTime 降順で返すため、1ページ目は
  最新時間帯のみで、本編・通勤・午前帯は2ページ目以降に押し出される。query を併用すると
  API独自スコア順となり、全日トップが先に来るため目的時間帯がさらに後ろへ流れる。
  どちらの場合も query / 時刻フィルタを当てにせず、全ページ走査で時系列マップを組む
- 推奨手順:
  1. query なし・時刻範囲なしで `limit=10` を投げ、nextCursor が無くなるまで全ページネーション
  2. 各ページから id/title/metadata.startTime/endTime のみ jq 抽出 → 時系列マップを作成
  3. 目的時間帯をカバーする lifelog を特定し、必要なものだけ transcript を取得
- 日付パラメータ (`date=YYYY-MM-DD`) は JST 0:00〜24:00 でしか効かない。日付窓と JST 日が
  一致しないケース(AM5:00 区切り等)や深夜帯まで活動が続くケースでは、対象日と翌日の
  **両日**を独立に全ページ走査して時系列マップをマージする
  * 片日だけ走査して「完走」と判定するのは禁止。深夜活動は翌日 date= 側にしか出ない
  * 両日それぞれの (全Nページ完走 / 最古・最新 startTime / 総件数) を作業メモに別々に記録

## レスポンス上限・トークン節約

- レスポンス上限超過時: 構造化テキストは `.[0].text` に JSON 文字列として入る。一時ファイル
  保存後 `jq '.[0].text | fromjson | .results'` で抽出。上限超過があってもページング自体は継続する
- トークン節約: 全文を context に流さない。メタデータ抽出 → 必要な lifelog のみ本文取得。
  キーワード違いで同じ時間帯を何度も query し直すと同一 lifelog が重複返却されるため、
  メタデータで特定した lifelog の id を起点に必要分だけ本文を取りに行く

## title の信頼性

- **lifelog の title を鵜呑みにしない**: title は Limitless 側で自動生成され、本文の
  一部だけを拾った要約に過ぎない。「会議の記録」「新しい仕事!」等の汎用タイトルが
  実は重要な内容を含むケースがある
- 目的時間帯に重なる lifelog は title だけで判別せず、必ず本文先頭 1〜2 ページ分を読んで
  内容を確認する

## 継ぎ目検出と「録音外」判定

- 継ぎ目検出: 取得済み lifelog の startTime/endTime で 10 分以上のギャップが見える時間帯
  は、追加ページネーション + 特徴的キーワードでの query 検索を再実行
- 「録音外」と確定するのは全ページネーション完走後のみ
  * 完走の根拠として取得した lifelog の (startTime 最古・最新・総件数) を作業メモに必ず
    残す。「完走」とだけ書いて根拠を残さないのは禁止
- 反パターン: 1ページ目だけで結論付ける(API は startTime 降順で返すため 1ページ目=最新
  時間帯のみ、本編・通勤・午前帯は2ページ目以降)。最古 startTime が活動の開始予想時刻
  より新しい間は完走扱いにしない
- 反パターン: 生活パターン推測(「装着なし」「疲労」「休日」「インドア」等)で結論付ける。
  判断材料は (全ページネーション後の件数・最古・最新 startTime) のみ
- 交差検証: 他ソース(自ツイート/GitHub/カレンダー等)で活動の痕跡があるのに lifelog
  件数=0 なら、ページネーション不足 or 検索条件不備をまず疑う

## 話者判定 (You / Unknown)

- `You` / `Unknown` ラベルは不安定、自分の発言も Unknown になることが多い
- 自分と推定する手がかり: `You` ラベル / 同時刻の自分の発信(ツイート等)と一致 / 自己紹介
  で自ハンドル・本名 / 外部ソースで主体が明らかな時間帯の発話
- Unknown 内の区別: 質問/回答の交互出現、自己紹介の流れ、文体や口調
- 特定できない発話は「参加者」「対話相手」等の匿名表記
- 原則: 確信なき人名補完なし / 自他不明なら自分に帰属させない / 複数人の発言を自分に
  誤帰属させない

## 音声認識の崩れと補正

- 補正対象は Limitless transcript のみ(他ソースの原文表記は尊重。意図的な
  スラング/俗字/略字との区別がつかないため)
- 起きやすい崩れ:
  * 濁音/半濁音の揺れ、促音・長音の脱落/付加
  * 単語境界の誤認、同音異義語、共通母音の漢字違い
  * 専門語が一般語化、地名/人名の片仮名化
  * 短音化されたカタカナや英単語の日本語化(プログラミング言語名、サービス名、プロダクト
    名、OS名、ツール名 等)
  * 日本語ビジネス用語のカタカナ化(approve系→承認、ringi系→稟議、compliance系→
    コンプライアンス)
- 検証手順: 同音近音候補を挙げる → 文脈と整合確認 → Web検索で実在確認 → 1つに絞れたら
  補正、複数残るなら削除/汎用化
- 強固に運用したい場合は上記を親が1パスで回さず、段を分けて並列化・相互検証する:
  * (a) 候補生成: 怪しいフレーズと音響近傍候補の列挙のみ。断定しない
  * (b) 検証: 各候補を Web 検索 + 文脈整合で潰す。フレーズ単位で並列化できるため
    サブエージェントに委譲も可能
  * (c) 相互検証: 同一チャンクを別根拠(音響近傍 vs 文脈整合)の2エージェントに投げ、
    結論が割れた箇所のみ親で再判定。1パス補正より誤訂正が減る
  * いずれの段でも確信度を「確定 / 候補複数 / 不能」の三値で扱い、確定以外は補正しない
- 補正できた場合は正解の表記だけを書く。原文(補正)のように両方を並記しない
- 意味不明フレーズ: 音写が激しく崩れていて裏付けからも特定できないフレーズは、そのまま
  残さず削るか、文脈を保てる汎用表現に置き換える
  * 行為の文脈が推定できる場合は汎用化(電話の呼び出し音らしい→「短い電話応対」、店員の
    挨拶らしい→「買い物」)
  * 行為の文脈も不明なら項目ごと削除
  * 「残すか削るか迷ったら削る」(読み手が文脈を掴めない情報は価値が低い)

## 認識アーティファクト

- 同一フレーズの異常反復 / 長大数字列 / 同一単語の連続行は捨てる
- 前後の文脈が取れない断片(独語・数字・感嘆詞)は捨てる。長短より「文脈が読めるか」で
  判断する
- 前後の文脈から意味が取れないカタカナ単語は削る。語数より「単独で意味が成立するか」で
  判断する
- 業界固有の略語・社内用語で意味が通じる場合はそのまま残してよい

## 視聴コンテンツ・アナウンスの扱い

- TV/ラジオ/動画/配信/ポッドキャスト/アニメ/ゲーム実況の視聴音声は本人の発言・反応を
  伴わない場合は要約対象外。詳細書き起こしは禁止、メタ情報(番組名・作品名・ジャンル・
  時間帯)のみ残す
- 駅・車内・店舗アナウンス等、自分の行動を伴わない放送のみの区間も要約対象外
  (「電車で帰宅」等の行動ラベルは残してよい)
- 例外: 地震・災害・遅延など本人の行動に影響した放送は内容も残す
